com.adventnet.nms.util.NmsSUM

NmsSUM  - Smart Update Manager, which takes care of the sync of the server & client. (i.e) Both are in the same product versions (For eg: both are in 4.7 SP1). If you are connecting a 4.7 application client to a 4.7 SP1 server you are prompted to whether install the ppm from the server or not. NmsSUM also ensures this sync when in case of Failover also. Commenting this is not recommended

com.adventnet.nms.store.DBUserStorageServer
From WebNMS developer guide

The cache size for user-defined objects (those persisted using UserStorageAPI) that are persisted in the database, can also be set in NmsProcessesBE.conf located in the <Web NMS Home>/conf directory.

For example, say you want to store 100 user-defined objects in memory. You can pass the argument 100 for the process com.adventnet.nms.store.DBUserStorageServer, as mentioned below :

PROCESS    com.adventnet.nms.store.DBUserStorageServer
ARGS       100
By default, a NULL value is specified, which implies that 25 user-defined objects will be stored in memory (This number is specified in the program DBUserStorageServer !) When there are no user defined objects to be persisted, this could be commented

com.adventnet.nms.startnms.AdminModuleInit
This is to get the server related information (like IP, various ports, FE details, client details) in the webclient. This could be commented out if you are not using webclient

com.adventnet.nms.sas.NMSSAServer & com.adventnet.nms.fe.sas.NmsSAServerFE

This service is to establish a contact between WebNMS with the devices. This process should be present in either (NmsProcessesBE / NmsProcessesFE) of the BE / FE side.
A detailed explanation is given at WebNMS developer guide

com.adventnet.nms.example.befe.ExampleFE

In the java client when you select the menu item Events And Alarms for a Snmp Node, you could see the details of Events & Alarms of that particular SNmp Node.
