Netconf Configuration Support in WebNMS

Overview

Netconf protocol based configuration support is added into WebNMS seamlessly integrating into the WebNMS Configuration module. This will allow the WebNMS framework to communicate with any Netconf supported device and retrieve, modify or delete configurations from the Management system. Each configuration is defined as a Configuration Task either through XML , Configuration API, or from a easy to use Graphical User Interface (GUI). The GUI allows the user to have full control on the configuration without the need to know the nitty-gritty of WebNMS Config XML formats as well as Netconf APIs or Protocol specifics. A sample Netconf based XML configuration template is shown below (This gets auto-generated from the GUI):


<?xml version="1.0" encoding="ISO-8859-1"?>
<ConfigTask taskName="Netconf_edit-config"
    description="NetConf edit-config merge operation"
    version="1.0"
    isTemplate="false"
    isNewTask="true"
    isSequential="false"
    isOverwrite="true"
    persistence="true"
    rollbackNeeded="false">

   <ProtocolMap name="netconf">
        <Device host="192.168.129.23" port="2022" />
   </ProtocolMap>

   <Attribute configOperation="edit-config"
             configAction="merge"
             defaultAction="merge"
              target="running"
              configPath="/hosts"
              subPath="host[name='Web-LB', enabled='false', numberOfServers='23']"
              namespace="http://acme.com/ns/simple/1.0" />

  <Attribute configOperation="edit-config"
             configAction="create"
             defaultAction="merge"
              target="running"
              configPath="/hosts"
              subPath="host[name='Web-Bkup', enabled='false', numberOfServers='2']"
              namespace="http://acme.com/ns/simple/1.0" />
</ConfigTask>



The above task definition aims at performing two configuration tasks:

1. Merging the attribute value changes in the Host Element ​"host[name='Web-LB', enabled='false', numberOfServers='23']"

2. Creating a new Host entry​ corresponding to ​"host[name='Web-Bkup', enabled='false', numberOfServers='2']"



Supported ​Netconf ​Configuration Operations​

WebNMS provides support to the following configuration operations:

  1. Get-Config

  2. Edit-Config

  3. Copy-Config

  4. Delete-Config

In addition, the Validate, Lock, Unlock and Commit are executed automatically as per the configurations provided in the task definition.

Get-Config Operation

The Netconf operation get-config allows the configuration to be retrieved from the device. The parameters allow the user to specify the target datastore, complete configuration or a filtered sub-config to be retrieved, etc.

Edit-Config Operation

The Netconf operation edit-config allows to edit an existing configuration in a target datastore. The actions such as creating new elements, changing element or attribute values, replacing exiting elements, or deleting elements are possible from this edit-config operation. The parameters to represent a edit-config operation in an WebNMS Netconf configuration XML task are listed below:


Task XML Attribute Name​

Sample Value​

Usage​

Comments​

configOperation

configOperation="edit-config"

The config operation defines the type of configuration that is going to be executed. Supported types are:

  1. edit-config

  2. copy-config

  3. delete-config

Edit-config is used to edit or change any of the configurations on a existing datastore.


configAction

configAction="merge"

This parameter defines the configuration action to be done. The permitted actions are:

  1. create

  2. merge

  3. replace

  4. delete

Create action is used when a new element is added to the target datastore.


Merge action is used whenever a attribute / Element value change is to be merged into an existing configuration.


Replace is used when an Element with same name is to be completely replaced with the given Element.


Delete is used when the given Element is to be deleted from the existing configuration.


defaultAction

defaultAction="merge"

Default action is taken when the configAction is not defined.


The allowed values are

  1. merge

  2. replace


target

target="running"

Netconf devices can support various configuration datastores such as :


1. Running

2. Startup

3. Candidate


So a configuration change via Netconf can be effected in any one of the above datastores.


The target parameter represents the datastore into which the configuration change needs to be effected.

The various supported datastores are published in the devices capabilities list.

configPath

configPath="/hosts"

This represents the root node of the device's configuration.

The root as well as allowed elements and attributes are listed in the device's schema definition

subPath

subPath="host[name='Web-LB', enabled='false', numberOfServers='23']"

This is the actual element being configured or changed.


namespace

namespace="http://acme.com/ns/simple/1.0"

This refers to the schema in which this configuration is defined.

A single device may implement various schema for various configuration.

errorOption

errorOption=”stop-on-error”

This parameter is used to indicate the action to be taken when an error is encountered while performing the configuration. The allowed values are:


  1. stop-on-error

  2. continue-on-error

  3. rollback-on-error

Rollback-on-error is supported only when the “rollback” capability is supported by the Netconf agent on the device.

testOption

testOption=”test-then-set”

TestOption allows the user to configuration needs to be validated before setting.

I.e if the Test-then-set value is used, the configuration change is first validated to check if it is proper before a configuration is set. The allowed values for this parameter are:

  1. test-then-set

  2. set




Copy-Config Operation

The Netconf operation copy-config allows the configurations to be copied from one datastore to another. Say for instance when a running configuration needs to be upgraded, the running configuration can be copied to the candidate configuration, then upgraded, then copied back to the running configuration. The parameters to represent a copy-config operation in an WebNMS Netconf configuration XML task are listed below:

Task XML Attribute Name

Sample Value

Usage

Comments

configOperation

configOperation="copy-config"

The config operation defines the type of configuration that is going to be executed. Supported types are:

  1. edit-config

  2. copy-config

  3. delete-config

The copy-config operation allows the configuration to be copied from one datastore to another.


Source / sourceURL

source=”running”

The source can be any valid supported datastores like :

  1. Startup

  2. Running

  3. Candidate

In case the configuration needs to be copied from a File or an URL then the sourceURL parameter can be used to indicate the file / url from which the configuration needs to be copied.


Target / targetURL

target=”candidate”

The target can be any of the valid supported datastores as above.

  1. Startup

  2. Running

  3. Candidate

In case the configuration needs to be copied from a File or an URL then the targetURL parameter can be used to indicate the file / url into which the configuration needs to be copied.






Delete-Config Operation

The Netconf operation delete-config is used for deleting an existing configuration or datastore. The parameters to represent a delete-config operation in an WebNMS Netconf configuration XML task are listed below:

Task XML Attribute Name

Sample Value

Usage

Comments

configOperation

configOperation="delete-config"

The config operation defines the type of configuration that is going to be executed. Supported types are:

  1. edit-config

  2. copy-config

  3. delete-config

The delete-config operation allows the configuration to be deleted from the give datastore.


Target / targetURL

target=”candidate”

The target can be any of the valid supported datastores as above.

  1. Startup

  2. Running

  3. Candidate

In case the configuration needs to be deleted from a File or an URL then the targetURL parameter can be used.

The configuration present in this datastore will be deleted.
