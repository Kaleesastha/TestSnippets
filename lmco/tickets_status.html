COMMS-339 3133749 Open WebNMS Ticket ID: ##3133749##: Investigate WebNMS behavior for handling MIB-II linkUp and linkDown traps (WebNMS Issue)
When testing the MIB-II interface linkUp and linkDown traps, we noticed the following behavior of WebNMS for processing the traps and generating corresponding event and alarm. It appears the “Failure Object” attribute of the generated event and alarm is not tied to the interface id in the trap. We think it should be fixed, so a unique alarm is generated and the cleared for each interface that has a failure and a trap is sent.
WebNMS implementation uses the trap filter (LinkUpDownTrapFilter.java) implementation to process the interface linkup and linkDown traps
An event is posted when a linkDown or linkUp trap is received. Only one alarm/clear is posted regardless of the interface associated with the link down/up trap.
- Alarm has a creation date and last update time. The last update time is updated with the time of the most recent trap received.
- Alarm text is updated with the most recent trap received, i.e. if multiple linkDown traps are received (e.g. linkDown for interface 1 first and interface next), the alarm text shows the text of the last one received. Same behavior for linkUp (clear) trap.
- linkUp trap for an interface on a node clears the alarm though the linkDown condition on another interface on the node still exists. This is because the alarm “Failure Object” (e.g. IF-hostname) for the link down/up alarm is associated to the node and not tied to the specific interface within the node for which the trap is sent.
Currently labeled as a non-issue. Confirm with WebNMS.

Sankar Ramaswamy
